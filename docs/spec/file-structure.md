## `.spec/` Directory

The `.spec/` directory is managed exclusively by `speqlite`. Do not edit files in this directory manually.

### `.spec/state.sqlite`

The canonical state database. Contains all spec nodes, relations, constraints, and event history.

- Created by `speqlite init`.
- Read by all commands.
- Written only by `speqlite apply`.
- Should be committed to Git for team sharing.
- Uses SQLite WAL mode for safe concurrent reads.

**Git recommendation**: Commit this file. SQLite WAL mode produces `-wal` and `-shm` files during active writes; add them to `.gitignore`.

```gitignore
.spec/state.sqlite-wal
.spec/state.sqlite-shm
```

### `.spec/state.snapshot.json`

A JSON snapshot of the metadata of all specs (no body text). Updated atomically by `speqlite apply`.

Purpose:
- Enables the planner to compute diffs without loading the full SQLite database.
- Is human-readable and Git-diffable.
- Used for conflict detection during `apply`.

**Git recommendation**: Commit this file. Review its diff in pull requests to see what specs changed.

Example:
```json
{
  "version": 1,
  "snapshot_hash": "sha256:cafebabe...",
  "taken_at": "2024-01-15T10:30:00Z",
  "specs": [
    {"id": "CMD-IMPORT", "title": "Import command", "kind": "command", "status": "fixed", "version": 3, "hash": "sha256:abc123...", "updated_at": "2024-01-15T10:30:00Z"}
  ],
  "relations": [
    {"from_id": "CMD-IMPORT", "relation": "depends_on", "to_id": "STATE-SQLITE"}
  ],
  "constraints": []
}
```

### `.spec/state.plan.json`

The pending plan, written by `speqlite import` and consumed by `speqlite apply`. Deleted after successful apply.

**Git recommendation**: Do **not** commit this file (add to `.gitignore`). Plans are ephemeral working state.

```gitignore
.spec/state.plan.json
```

If you want to preserve plans for audit, use the `changes/` directory (see below).

## `specs/` Directory

Contains Markdown projections rendered by `speqlite render`. These files are derived from state and can always be regenerated.

### File Naming

```
{spec-id}.md
```

Examples: `CMD-IMPORT.md`, `FR-001.md`, `STATE-SQLITE.md`

### File Format

```markdown
# CMD-IMPORT: Import Command

**Kind**: command
**Status**: fixed
**Version**: 3
**Updated**: 2024-01-15T10:30:00Z

## Description

The import command reads Markdown or plain-text files...

## Relations

- depends_on: [STATE-SQLITE](STATE-SQLITE.md)
- implements: [FR-001](FR-001.md)

---
<!-- speqlite:id=CMD-IMPORT speqlite:version=3 speqlite:hash=sha256:abc123 -->
```

The HTML comment at the end is the *roundtrip marker*. The normalizer uses it to match rendered files back to their canonical IDs without relying on filenames.

**Git recommendation**: Commit these files. They serve as human- and AI-readable documentation.

---

## `scratch/` Directory

A working area for rough input files. Users write informal Markdown or plain text here, then import it.

Convention:
- Files in `scratch/` are not managed by Speclite.
- They are free-form; the normalizer handles them.
- After import and apply, scratch files can be archived or deleted.

**Git recommendation**: Optional. Some teams commit scratch files as a record of intent; others `.gitignore` them.

---

## `changes/` Directory

An optional directory for archiving applied plans. After `speqlite apply` succeeds, users can move (or configure Speclite to copy) `state.plan.json` here with a timestamp prefix.

```
changes/
  20240115T103000Z-apply.json
  20240116T082000Z-apply.json
```

This provides a full audit trail of every state mutation, even beyond what the event log records (which stays in SQLite).

**Git recommendation**: Commit if you want a Git-tracked audit trail outside SQLite.

```gitignore
# Speclite ephemeral files

```

## File Ownership Summary

| Path | Managed by | Git |
|---|---|---|
| `.spec/state.sqlite` | `speqlite apply` only | Commit |
| `.spec/state.snapshot.json` | `speqlite apply` only | Commit |
| `.spec/state.plan.json` | `speqlite import` / `apply` | Ignore |
| `.spec/state.sqlite-wal` | SQLite | Ignore |
| `.spec/state.sqlite-shm` | SQLite | Ignore |
| `specs/*.md` | `speqlite render` | Commit |
| `scratch/*` | User | Optional |
| `changes/*.json` | User / scripts | Optional |

---

## Multi-Repository Workspaces

Speclite is designed for single-repository use in the MVP. If you have multiple repositories, run `speqlite init` in each and use `speqlite state export` to exchange snapshots.

Cross-repository relation tracking (where `from_id` and `to_id` live in different workspaces) is a post-MVP feature.

## Environment Variable Overrides

| Variable | Default | Description |
|---|---|---|
| `SPECLITE_WORKSPACE` | `.` | Workspace root directory |
| `SPECLITE_DB_PATH` | `.spec/state.sqlite` | Path to SQLite database |
| `SPECLITE_PLAN_PATH` | `.spec/state.plan.json` | Path to plan file |
| `SPECLITE_SNAPSHOT_PATH` | `.spec/state.snapshot.json` | Path to snapshot file |
| `SPECLITE_SPECS_DIR` | `specs/` | Directory for rendered Markdown |
| `SPECLITE_SCRATCH_DIR` | `scratch/` | Directory for input files |

All environment variables can be overridden by the corresponding CLI flag (`--workspace`, `--db-path`, etc.).
