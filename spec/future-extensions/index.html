    Future Extensions¶

    This page documents the planned extension points for Speclite beyond the MVP. These are first-class architectural concerns — the data model and state machine are designed to support them — but they are not implemented in v0.1.

    MCP Server¶

    The Model Context Protocol (MCP) allows AI assistants to call structured tools to retrieve context. A Speclite MCP server exposes spec state as an AI-readable API without requiring the AI to run CLI commands or parse raw Markdown.

    Motivation¶

    In an AI-native workflow, the LLM needs: - Access to the latest stable specifications - The ability to search by topic - The ability to traverse dependency graphs - A way to check if a proposed change would violate constraints

    The MCP server provides all of this as structured function calls.

    Proposed Tool Interface¶

    spec_get¶

    Retrieve a single spec node by ID.

    {
  "name": "spec_get",
  "description": "Get a single specification by ID",
  "inputSchema": {
    "type": "object",
    "properties": {
      "id": {
        "type": "string",
        "description": "Spec ID, e.g. CMD-IMPORT"
      },
      "include_body": {
        "type": "boolean",
        "default": true,
        "description": "Include full body text in response"
      }
    },
    "required": ["id"]
  }
}

    Response: 

    {
  "id": "CMD-IMPORT",
  "title": "Import command",
  "kind": "command",
  "status": "fixed",
  "version": 3,
  "body": "The import command reads...",
  "relations": [
    {"relation": "depends_on", "to_id": "STATE-SQLITE"},
    {"relation": "implements", "to_id": "FR-001"}
  ]
}

    spec_search¶

    Full-text search over specs using FTS5/BM25.

    {
  "name": "spec_search",
  "description": "Search specs by keyword",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {"type": "string"},
      "limit": {"type": "integer", "default": 10},
      "kind": {"type": "string"},
      "status": {"type": "string"}
    },
    "required": ["query"]
  }
}

    spec_deps¶

    Traverse the dependency graph from a root node.

    {
  "name": "spec_deps",
  "description": "Get the dependency graph for a spec",
  "inputSchema": {
    "type": "object",
    "properties": {
      "id": {"type": "string"},
      "direction": {"type": "string", "enum": ["out", "in", "both"], "default": "out"},
      "depth": {"type": "integer", "default": 3},
      "relation": {"type": "string"}
    },
    "required": ["id"]
  }
}

    spec_validate¶

    Run validation checks and return structured errors.

    {
  "name": "spec_validate",
  "description": "Validate the current spec state",
  "inputSchema": {
    "type": "object",
    "properties": {
      "spec_id": {"type": "string", "description": "Validate only this spec; omit for full workspace"},
      "strict": {"type": "boolean", "default": false}
    }
  }
}

    spec_list¶

    List specs matching filters.

    {
  "name": "spec_list",
  "description": "List specs with optional filtering",
  "inputSchema": {
    "type": "object",
    "properties": {
      "kind": {"type": "string"},
      "status": {"type": "string"},
      "limit": {"type": "integer", "default": 50}
    }
  }
}

    MCP Server Architecture¶

    speqlite mcp serve [--port 3000] [--transport stdio|sse]
    • Default transport: stdio (for use with Claude Desktop, Cursor, etc.)
    • Optional transport: sse (Server-Sent Events for HTTP clients)
    • Reads from state.sqlite in read-only mode (no mutations via MCP in v1)
    • Authentication: none in MVP (local-only); API key in future versions

    Implementation Notes¶

    • Use the mark3labs/mcp-go library for the Go implementation.
    • The MCP server is a separate binary target: cmd/speqlite-mcp/main.go.
    • It shares the internal/db and internal/search packages with the CLI.

    Formal Language Export¶

    The constraints table stores formal expressions for future export to verification tools. The export pipeline emits these expressions in tool-specific syntax.

    Supported Export Targets¶

    Alloy¶

    Alloy is a relational modelling language for structural invariants.

    speqlite export --format alloy --output spec.als

    MVP output (stub): 

    -- Generated by speqlite export --format alloy
-- Workspace: my-project
-- Date: 2024-01-15

module Speclite

sig Spec {
  id: one String,
  kind: one Kind,
  status: one Status,
  depends_on: set Spec,
  implements: set Spec
}

enum Kind { command, requirement, architecture, state, constraint, test, glossary, note }
enum Status { draft, review, fixed, implemented, verified, deprecated }

-- Constraints from workspace:
-- (full expression generation is post-MVP)

    SMT-LIB 2¶

    For constraint solving with Z3 or CVC5.

    speqlite export --format smtlib --output spec.smt2

    MVP output (stub): 

    ; Generated by speqlite export --format smtlib
; Workspace: my-project

(declare-sort Spec 0)
(declare-fun id (Spec) String)
(declare-fun kind (Spec) String)
(declare-fun status (Spec) String)
(declare-fun depends_on (Spec Spec) Bool)

; Constraints: (post-MVP)

    TLA+¶

    For model checking with TLC.

    speqlite export --format tla --output spec.tla

    MVP output (stub): 

    ---- MODULE Speclite ----
EXTENDS Sequences, FiniteSets

CONSTANTS SpecIDs

VARIABLES status

TypeInvariant ==
  \A id \in SpecIDs :
    status[id] \in {"draft","review","fixed","implemented","verified","deprecated"}

====

    Lean 4¶

    For theorem proving.

    speqlite export --format lean --output Spec.lean

    Datalog¶

    For deductive reasoning with Soufflé or µZ.

    speqlite export --format datalog --output spec.dl

    Export Pipeline Architecture¶

    internal/exporter/
  alloy.go      -- Alloy .als generator
  smtlib.go     -- SMT-LIB 2 .smt2 generator
  tla.go        -- TLA+ .tla generator
  lean.go       -- Lean 4 .lean generator
  datalog.go    -- Datalog .dl generator
  exporter.go   -- common interface: Exporter.Export([]SpecNode, io.Writer) error

    The Exporter interface:

    type Exporter interface {
    Format() string
    Export(nodes []db.Spec, relations []db.Relation, constraints []db.Constraint, w io.Writer) error
}

    Constraint Expression Language (Future)¶

    In a future version, constraints will be authored in a Speclite DSL that compiles to multiple targets:

    CONSTRAINT CONSTR-001 ON FR-001:
  INVARIANT: all specs in status "implemented" have at least one "verifies" relation

    This DSL is not part of the MVP. MVP constraints are stored as raw strings in the target language and emitted verbatim.

    Additional Post-MVP Features¶

    Web UI (non-goal until v2)¶

    A local web server (speqlite ui) serving a React SPA that visualises the spec graph. Not in scope for v1.

    Watch Mode¶

    speqlite watch scratch/

    Automatically re-imports changed files and updates the pending plan. Implemented as a file-system watcher loop.

    CI Integration¶

    speqlite validate --exit-on-error

    Returns exit code 2 on validation errors, suitable for use in CI pipelines. This is already part of the MVP CLI but documented here for CI-specific usage patterns.

    Spec Linting¶

    Custom lint rules defined in .spec/lint.yaml:

    rules:
  - id: no-draft-fixed-dependency
    level: error
  - id: require-body-for-fixed
    level: warning
  - id: max-depth
    value: 5
    level: warning

    Embedding / Semantic Search¶

    Integration with a local embedding model (e.g., nomic-embed-text via Ollama) to provide semantic search alongside BM25. Requires a vector storage extension — post-MVP.
